# تقرير تحليل منصة "ترتيلة أونلاين" — الميزات والتعقيد والتسعير التجاري

**تاريخ التقرير:** 2026-07-05
**نوع التقرير:** تحليل فني وتجاري شامل، مبني على فحص فعلي للكود المصدري (Frontend + Backend + قاعدة البيانات)، وليس على افتراضات أو استنتاجات من الواجهات فقط.

## 1. الملخص التنفيذي (Executive Summary)

"ترتيلة أونلاين" (Tartelah Online) هي منصة تعليمية متكاملة لتحفيظ وتعليم القرآن الكريم عن بُعد، مبنية بتقنية MERN (MongoDB + Express + React + Node.js)، بثلاثة أدوار مستخدمين رئيسية: **طالب**، **معلم**، **مدير (Admin)** — دون وجود دور "ولي أمر" فعلي في النظام، على الرغم من أن النصوص التسويقية تَعِد بتقارير تصل لأولياء الأمور.

المنصة **ليست نموذجًا أوليًا (Prototype)** ولا واجهة بدون محتوى حقيقي — 100% من صفحات لوحات التحكم الثلاث (الطالب، المعلم، المدير) متصلة فعليًا بواجهات API حقيقية عبر React Query + Axios، مع نظام مصادقة JWT حقيقي (Access + Refresh Token)، وقاعدة بيانات MongoDB بها 21 موديل بعلاقات ومنطق أعمال حقيقي. حجم الكود المصدري الفعلي (بدون المكتبات الخارجية) يبلغ **حوالي 36,500 سطر** (8,456 سطر Backend + 28,006 سطر Frontend)، موزّعة على **104 ملف Backend** و **112 ملف Frontend** — وهو حجم يضع المشروع في فئة "منتج SaaS متوسط إلى كبير"، وليس مشروع تخرج أو MVP بسيط.

النقطة الأهم لفهم القيمة التجارية الحقيقية للمنصة: **لا يوجد أي نظام دفع إلكتروني حقيقي (Payment Gateway)** — لا Stripe ولا Paymob ولا Fawry ولا PayPal. آلية "الدفع" الكاملة هي: الطالب يختار باقة، يُعلن طريقة دفع (تحويل بنكي/كاش) كنص فقط، يرفع صورة إثبات دفع، ثم يوافق المدير يدويًا وينشئ اشتراكًا. هذه معلومة جوهرية لأي تسعير تجاري، لأنها تعني أن الجزء الأكثر تكلفة عادة في منصات SaaS التعليمية (بوابات الدفع والفوترة المتكررة) **غير موجود إطلاقًا**.

كذلك، **لا يوجد تكامل فيديو حقيقي** (لا Zoom API ولا Google Meet API ولا WebRTC) — "الحصة المباشرة" هي مجرد رابط خارجي (URL) يُلصَق يدويًا، مع نظام ذكي جدًا لتتبّع الحضور والراتب يعتمد على "بصمات زمنية" (توقيت تسجيل الدخول، توقيت فتح الرابط) لا على تحقق فعلي من دخول الاجتماع — وهذا موثّق بوضوح ونزاهة في تعليقات الكود نفسه.

في المقابل، المنصة تحتوي على **أنظمة داخلية معقدة وذكية بُنيت بالكامل من الصفر** وتُعتبر أهم ما يرفع قيمتها التجارية:
- محرك جدولة متكرر (Recurring Scheduling Engine) يولّد حصصًا فعلية من قواعد جدولة، بحماية من التكرار.
- محرك "ذكاء الحضور والرواتب" (Session Intelligence) يحسب حالة الدفع لكل حصة (قابلة للدفع / مرفوضة / تحت المراجعة) بقواعد صريحة وموثّقة، مع درجة "ثقة" لكل حصة.
- "مركز عمليات" (Operations Center) كامل بثلاث تبويبات (الآن / الجدول الزمني / قائمة المراجعة) لإدارة الاستثناءات التشغيلية — وهي من أكثر الميزات المخصّصة تعقيدًا في المنصة.
- نظام تدقيق (Audit Log) حقيقي يغطي معظم — وليس كل — العمليات الحساسة.
- نظام إشعارات فورية (Real-Time) عبر Socket.io — لكنه **معطّل حاليًا بسبب خطأ في متغيرات البيئة** (تفصيل في القسم 17).
- مساعد ذكاء اصطناعي حقيقي (Claude Haiku عبر Anthropic API) مع نظام بحث داخل المقالات (RAG) وخطة احتياطية بقاعدة معرفة ثابتة — لكنه **غير موصول بأي رابط تنقّل في الواجهة** (ميزة "مخفية" فعليًا لا يصل إليها أي مستخدم عادي).

**التقييم العام للتعقيد: 7.5 من 10** (تفصيل الأسباب في القسم 19).

## 4. تدفقات المستخدم الكاملة (User Flows)

### تدفق التسجيل والالتحاق (الأهم تجاريًا)
```
زائر يتصفّح الموقع → يختار معلمًا أو باقة → يضغط "تسجيل"
  → ينشئ حساب طالب (الدور مُثبَّت "طالب" دائمًا، لا اختيار دور، لا تفعيل بريد)
  → يدخل لوحة تحكم الطالب → صفحة "طلب الالتحاق"
  → يختار باقة (من جدول Package الحقيقي، بأسعار بالريال السعودي)
  → يُعلن طريقة دفع (تحويل بنكي/كاش — نص فقط، بلا معالجة فعلية)
  → يرفع صورة إثبات تحويل (حتى 5MB، jpg/png/webp)
  → الطلب يدخل حالة pending → under_review
  → المدير يفتح "مراجعة طلبات الالتحاق"، يشاهد صورة الإثبات، يوافق ويُعيّن معلمًا يدويًا
  → يُنشأ Subscription بحالة active + يُعبّأ amountPaid برقم Package.price (بلا أي تحقق مصرفي فعلي)
  → إشعار فوري (لو كان يعمل) للطالب والمعلم
```

### تدفق الحصة الدراسية الكاملة
```
المعلم يفتح "الحصص والجدولة" → معالج جدولة متكرر (4 خطوات: طالب → تكرار وأيام ووقت → عدد حصص ورابط → معاينة تواريخ)
  → Backend يولّد جميع الحصص فعليًا (بحماية من التكرار عبر Index فريد)
  → في يوم الحصة: المعلم يضغط "تسجيل الحضور وفتح الفصل"
      → يُسجَّل توقيت دخول المعلم (يحدد "في الوقت/متأخر" مقابل الوقت المجدول)
      → يُفتح رابط Zoom/Meet الخارجي في تبويب جديد (بلا أي تكامل API فعلي)
  → بعد الحصة: المعلم يسجّل حضور الطالب (حاضر/غائب/متأخر/معذور) ويُنهي (Finalize) السجل
  → المعلم (اختياريًا) يفتح تقييم سريع أو واجب سريع مرتبط بالحصة نفسها
  → إن لم يتحرّك أحد: مهمة Cron كل 10 دقائق تُصعِّد الحصة تدريجيًا (scheduled → missed → no_show) مع حساب حالة الدفع تلقائيًا (إلا إن صحّحها المدير يدويًا سلفًا)
  → محرك "ذكاء الحصة" يحسب درجة ثقة وحالة راتب (قابلة للدفع/مرفوضة/تحت المراجعة) لكل حصة، تراها لوحة المعلم ومركز عمليات المدير
```

### تدفق المساعد الذكي (ميزة يتيمة فعليًا)
```
المستخدم (أي دور) → يجب أن يكتب /ai في شريط العنوان يدويًا (لا يوجد رابط تنقّل في أي قائمة)
  → يسأل سؤالًا عن التجويد/الحفظ → Backend يبحث في المقالات المنشورة (بحث نصّي) + يستدعي Claude Haiku (لو مفتاح API موجود)
  → إن لم يوجد مفتاح API: يرجع لإجابة ثابتة من قاعدة معرفة محدودة (~8 قواعد تجويد + معلومات أكاديمية)
```

## 5. إمكانات المدير (Admin) بالتفصيل

المدير يملك أكبر سطح تحكم في المنصة (22 صفحة)، ويمكن تلخيص إمكاناته في 6 مجموعات رئيسية:

1. **إدارة المستخدمين (CRM كامل):** بحث وفرز وتصفية للطلاب والمعلمين، لوحات جانبية منزلقة (Drawer) بتبويبات (معلومات/أداء/تعديل/إعادة تعيين كلمة مرور)، تفعيل/تعطيل (حذف ناعم فقط، لا حذف نهائي)، إجراءات سريعة (اتصال/واتساب/بريد مباشرة).
2. **الإدارة الأكاديمية:** مراجعة طلبات الالتحاق مع عارض صور إثبات الدفع، إدارة الاشتراكات والباقات، إدارة قواعد الجدولة والحصص مع تصحيح حضور/راتب مباشر من الجدول.
3. **مركز العمليات (الميزة الأكثر تخصيصًا):** لوحة تشغيلية حية بثلاث تبويبات — "الآن" (حصص حية/متأخرة/بلا رابط اجتماع/بانتظار مراجعة)، "الجدول الزمني" (سجل قابل للتصفية بتاريخ/معلم/حالة)، و"قائمة المراجعة" (طابور حالات استثنائية مُرتَّب بالخطورة مع نموذج تصحيح مباشر). هذا نظام مبني بالكامل خصيصًا لهذه المنصة، وليس قالبًا جاهزًا.
4. **التقارير والتحليلات وسجل التدقيق:** لوحة KPIs (إيرادات/طلاب/حصص) حقيقية الأرقام لكن **الرسوم البيانية للاتجاهات الشهرية مُصنَّعة رياضيًا** من رقم إجمالي واحد مُوزَّع على أوزان ثابتة (تفصيل في القسم 17)، مع سجل تدقيق حقيقي يغطي معظم — وليس كل — العمليات الحساسة.
5. **إدارة المحتوى (CMS):** نظام مقالات/مدونة كامل (محرر نصوص غني مبني يدويًا، فئات، جدولة نشر، حذف ناعم، بحث نصّي)، إدارة قصص النجاح (وضعان: بطاقات/بانر)، إدارة آراء العملاء والأسئلة الشائعة وإعدادات الموقع العامة — **لكن** أغلب صفحات الموقع التسويقي (الرئيسية، الأسعار، من نحن) لا تستهلك فعليًا هذا الـ CMS (محتواها مكتوب مباشرة في الكود).
6. **إدارة أداء ورواتب المعلمين:** لوحة قيادة (Leaderboard) لكل المعلمين بمقاييس الانضباط والاكتمال والغياب، مع تصدير Excel/CSV وتصدير PDF حقيقي (عبر html2canvas + jsPDF، مصمم خصيصًا لدعم النص العربي).

## 6. إمكانات المعلم بالتفصيل

- **الجدولة الذكية:** معالج بناء جدول متكرر بـ4 خطوات مع معاينة تواريخ من الخادم قبل الحفظ، تقويم شهري للحصص، بطاقات حصص قابلة للتوسيع بكل الإجراءات (حضور، إكمال، تأجيل، إلغاء، تقييم سريع، واجب سريع) داخل البطاقة نفسها.
- **الحضور الذاتي:** زر واحد "تسجيل الحضور وفتح الفصل" يسجّل توقيت دخول المعلم (نظام تسامح متدرّج لا حدّ صارم واحد) ويفتح الرابط الخارجي — موثّق في الكود بوضوح أنه **لا يُثبت الحضور الفعلي في الاجتماع**.
- **تقييم الطلاب:** تقييمات مُصنَّفة (تجويد/حفظ/نظرة/سلوك/عام) من 10، مع نقاط قوة ونقاط تحتاج تحسين، وخيار مشاركتها مع الطالب أو إخفائها.
- **الواجبات:** إنشاء واجبات موجّهة لطالب أو أكثر، وتصحيح تسليمات الطلاب (درجة + ملاحظات) — لا يوجد رفع ملفات من طرف المعلم نفسه في هذه الصفحة (فقط عرض/تصحيح نص).
- **تتبّع التقدّم:** تسجيل حفظ ومراجعة بمستوى (سورة + مدى آيات + تقييم جودة).
- **مكتبة الروابط الدائمة:** حفظ روابط اجتماعات شخصية (Zoom/Meet/Teams) لإعادة استخدامها بلا الحاجة لإنشاء رابط جديد كل مرة.
- **الأداء والراتب الشخصي:** رسوم بيانية (توزيع الحضور، اتجاه أسبوعي/شهري)، إجمالي المستحق مالياً، تفاصيل الجلسات القابلة للدفع/غير القابلة، تصدير CSV وPDF.

## 7. إمكانات الطالب بالتفصيل

- **لوحة رئيسية تحفيزية:** عدّاد تنازلي مباشر للحصة القادمة، حلقة تقدّم بصرية لنسبة الحفظ، "آية اليوم" (دورانية محلية غير مرتبطة بالخادم)، مهام اليوم.
- **الالتحاق بإثبات دفع:** معالج بخطوات (اختيار باقة → إعلان طريقة دفع → رفع صورة إثبات) مع تتبّع حالة الطلب.
- **الحصص والجدول:** عرض الحصص القادمة والسجل، انضمام بضغطة رابط خارجي واحدة.
- **الواجبات:** تسليم فعلي بملفات مرفقة (حتى 3 ملفات، 20MB للملف)، مع رؤية الدرجة والملاحظات بعد التصحيح.
- **متابعة التقدّم:** سجلّان منفصلان (حفظ/مراجعة) بمستوى سورة ومدى آيات.
- **الإشعارات:** مركز إشعارات كامل بفلاتر وبحث وتحديد جماعي.
- **الاشتراك:** عرض حالة الاشتراك وأيام متبقية — **قراءة فقط**، أي تجديد يوجّه مباشرة لصفحة طلب التحاق جديد (لا تجديد ذاتي مباشر).

## 8. إمكانات ولي الأمر

**غير موجودة إطلاقًا.** تأكيد نهائي عبر فحص شامل للكود (Backend + Frontend):
- `User.role` في قاعدة البيانات محدود بثلاث قيم فقط: `admin | teacher | student` — لا توجد قيمة "parent".
- لا يوجد أي Model أو Route أو Controller أو صفحة مرتبطة بمفهوم "ولي أمر" في أي مكان بالكود.
- **تحذير تجاري مهم:** صفحتا "آراء الطلاب" (Testimonials) و"من نحن" (About) تحتويان على نصوص تسويقية صريحة تَعِد بأن "تقارير أداء دورية تصل لأولياء الأمور" — وهذا **وعد تسويقي بلا أي غطاء تقني فعلي**، ما يمثّل خطر مصداقية إذا نُشر الموقع كما هو دون بناء الميزة أو تصحيح النص.

## 10. تعقيد الـ Backend والـ API

- **21 موديل بيانات، 21 Controller، 24 ملف Routes.**
- نظام صلاحيات RBAC حقيقي مطبَّق على مستوى الـ Router في 20 من 24 ملف Routes.
- التحقق من صحة المدخلات (Validation) عبر Joi **مطبَّق فقط على 2 من 24 ملف Routes** (المصادقة والواجبات) — بقية الـ 22 ملف تعتمد فقط على تحقق Mongoose الأساسي (أضعف، بلا رفض حقول غير متوقعة أو تحقق متعدد الحقول). هذه نقطة تحتاج تحسين أمني/جودة حقيقية.
- محرك جدولة متكرر حقيقي (`schedule.service.js`) بحماية Index فريد من تكرار الحصص التلقائية.
- محرك "ذكاء الحصة" (`sessionIntelligence.service.js`) — قواعد حتمية موثّقة تحسب حالة الدفع ودرجة الثقة، مغطّى باختبارات وحدة حقيقية.
- 3 مهام Cron حقيقية تعمل تلقائيًا (تذكير حصص، انتهاء اشتراكات، تتبّع غياب تدريجي).
- نظام إشعارات مزدوج (REST للتاريخ/الإدارة + Socket.io للحظة الفورية) — لكن الجزء الفوري **معطّل حاليًا بسبب خطأ تهيئة** (القسم 17).
- سجل تدقيق (Audit Log) حقيقي يغطي نطاقات الحصص/الحضور/الرواتب/الاشتراكات/الالتحاق/المقالات — لا يغطي عمليات تسجيل الدخول أو إدارة المستخدمين العامة أو الدورات أو الباقات.

## 11. تعقيد قاعدة البيانات

نموذج `Session` هو الأكثر تعقيدًا في كامل قاعدة البيانات: يحتوي على أكثر من 25 حقلًا (حالة، حضور المعلم، توقيتات فعلية مقابل مجدولة، تأخير، نتيجة الحصة، حالة الدفع مع من ضبطها، دورة مراجعة كاملة)، مع 8 فهارس (Indexes) بما فيها فهرس فريد جزئي لمنع تكرار الحصص المُولَّدة تلقائيًا. هذا مستوى تصميم بيانات متقدّم وليس نموذجًا بسيطًا.

نقطة معمارية لافتة: يوجد **نموذجان للالتحاق منفصلان** — `Enrollment` (قديم، يربط طالبًا بدورة، لكنه **لا يُكتَب فيه فعليًا من أي Controller حاليًا**، فقط يُقرأ وتُعبّئه بيانات البذر التجريبية Seed) و`EnrollmentRequest` (الفعلي المُستخدَم، يربط طالبًا بباقة). هذا يعني أن **الدورات (Courses) غير مرتبطة فعليًا بخط الدفع/الاشتراك/الحصص** — الدورات تعمل كمحتوى تسويقي/كتالوج مستقل، والوصول الفعلي للمعلم والحصص يتم بالكامل عبر مسار الباقات (Package)، وهذا يُنشئ ازدواجية مفهومية تحتاج توضيحًا لأي عميل يعتقد أن "شراء دورة" يُفعِّل حصصًا مباشرة.

## 12. تعقيد واجهة المستخدم/تجربة المستخدم (UI/UX)

- **دعم RTL حقيقي ومتعمَّد**: `dir="rtl"` مضبوطة عالميًا، ومعاد تأكيدها يدويًا في 120 موضعًا عبر 64 ملفًا (دفاعيًا)، مع استخدام خصائص CSS منطقية (`insetInlineEnd`، `marginInlineStart`) بدل `left/right` المباشرة — هذا مستوى احترافي حقيقي في هندسة الاتجاه العربي، وليس ترجمة سطحية.
- **استراتيجيتان متجاوبتان متوازيتان**: الموقع التسويقي مبني بـ`clamp()` يدوية (93 استخدامًا في 16 ملفًا) و`@media` مخصّصة (13 استخدامًا)، بينما لوحات التحكم الثلاث تعتمد على فئات Tailwind القياسية (218 استخدامًا في 50 ملفًا).
- مكوّنات UI متقدّمة حقيقية: محرر نصوص غني مبني يدويًا بالكامل (لا مكتبة خارجية مثل TipTap)، معالج جدولة متكرر بخطوات وتقدّم مرئي، محرر منهج دورة ديناميكي (أقسام ودروس)، رفع صور بالسحب والإفراغ مع اقتصاص (`react-easy-crop`)، لوحات جانبية منزلقة (CRM Drawers)، تصدير PDF حقيقي بدعم عربي كامل عبر تحويل جدول HTML لصورة (بسبب قصور jsPDF في عرض الخطوط العربية مباشرة) ثم تحويلها لـPDF.
- تحريك (Animation) مكثّف عبر Framer Motion في كل الصفحات التسويقية والعديد من صفحات لوحة التحكم.

## 14. الأمان والمصادقة

**النقاط القوية المؤكدة:**
- JWT مزدوج (Access قصير الأجل + Refresh عبر Cookie httpOnly) مع تجديد تلقائي حقيقي عبر Axios Interceptor (نمط إنتاجي صحيح، ليس تجريبيًا).
- تجزيء كلمات المرور بـ bcrypt (12 دورة).
- استعادة كلمة مرور حقيقية بالكامل (رمز مُشفَّر SHA-256، صلاحية 30 دقيقة، بريد فعلي).
- RBAC مطبَّق على مستوى الراوتر في أغلب الملفات.
- Helmet + CORS محدود بأصل واحد + Rate Limiting (عام + أشد على المصادقة تحديدًا).
- سجل تدقيق حقيقي (جزئي التغطية).
- بوابة "دخول تجريبي" (Dev Login) محمية بشكل صحيح فنيًا (تُرجع 404 في بيئة الإنتاج).

**الثغرات والقيود المكتشفة:**
- **لا يوجد تفعيل بريد إلكتروني (Email Verification)** — الحقل موجود بقاعدة البيانات (`isEmailVerified`) لكن لا يوجد أي مسار فعلي يُفعِّله أو يتحقق منه.
- **لا يوجد مصادقة ثنائية (2FA)** بأي شكل.
- **لا يوجد تسجيل دخول اجتماعي** (Google/Facebook).
- **لا يوجد اختيار دور عند التسجيل** — كل تسجيل جديد يُثبَّت كـ"طالب" فقط، بلا واجهة لإنشاء حساب معلم من الموقع.
- **مجلد `/uploads` بالكامل متاح للعامة بلا أي حماية مصادقة** — أي شخص يعرف/يخمّن اسم الملف يمكنه الوصول لصور إثبات الدفع المالي (خصوصية حساسة).
- **التحقق من المدخلات (Validation) ضعيف في 22 من 24 مسار API** — يعتمد فقط على تحقق Mongoose الأساسي.
- **لا يوجد إبطال جلسة فعلي عند تسجيل الخروج (Session Revocation)** — حقل `refreshToken` موجود بالنموذج لكن لا يُكتَب فيه فعليًا، فلا توجد قدرة "إنهاء كل الجلسات" من الخادم.

## 16. الميزات التي ترفع التكلفة بشكل كبير

هذه هي الميزات التي — بحسب فحص الكود — تمثّل أكبر استثمار هندسي حقيقي في المنصة، وتُبرِّر جزءًا كبيرًا من السعر:

1. **محرك ذكاء الحضور والرواتب + مركز العمليات** — نظام قواعد حتمية مخصّص بالكامل، غير موجود في أي قالب جاهز، ومغطّى باختبارات.
2. **محرك الجدولة المتكررة** — توليد حصص من قواعد متكررة بحماية من التكرار، مع معاينة من الخادم قبل الحفظ.
3. **نظام CMS كامل للمقالات** — محرر نصوص غني مبني يدويًا، بحث نصّي، جدولة نشر، إعجاب/حفظ، حساب وقت قراءة تلقائي.
4. **CRM كامل للطلاب والمعلمين** — لوحات جانبية بتبويبات متعددة، بحث/فرز/تصفية، إجراءات سريعة.
5. **نظام تصدير PDF عربي حقيقي** — حلّ تقني غير تافه (Rasterize HTML → PDF) بسبب قصور المكتبات الجاهزة في عرض العربية.
6. **62+ صفحة/مسار موزّعة على 3 لوحات تحكم بأدوار مختلفة**، كل واحدة بحمايتها ومنطقها الخاص.

## 17. الميزات الناقصة أو غير المكتملة

### أ) ميزات مُعلَن عنها تسويقيًا لكن غير موجودة فعليًا (خطر مصداقية)
- صفحة الأسعار تَعِد بدعم بطاقات ائتمان وPayPal و"دفع آمن 100%" — **لا شيء من هذا موجود بالكود**.
- نصوص تسويقية تَعِد بتقارير لأولياء الأمور — **لا يوجد دور ولي أمر إطلاقًا**.
- المساعد الذكي يُسوَّق كميزة خاصة بالباقات الأعلى (Pro/Elite) — **بلا أي تحقق فعلي من الباقة في الكود**، وأي مستخدم مسجّل يمكنه الوصول له لو عرف الرابط.

### ب) أخطاء وتناقضات حقيقية مكتشفة
- **الإشعارات الفورية معطّلة**: متغير بيئة `JWT_SECRET` غير مُعرَّف (النظام يستخدم أسماء مختلفة في كل مكان آخر)، فكل اتصالات Socket.io تُرفض حاليًا.
- **صفحة الأسعار العامة منفصلة تمامًا عن نظام الباقات الحقيقي**: أرقام وعملة مختلفة (دولار بدل ريال)، وأزرارها تذهب للتسجيل فقط لا لأي شراء فعلي.
- **نظامان منفصلان للأسئلة الشائعة**: واحد ثابت بالكود (تستخدمه صفحة FAQ المستقلة)، وآخر حقيقي بقاعدة بيانات (تستخدمه صفحة التواصل فقط) — لا تناسق بينهما.
- **صفحة المساعد الذكي "يتيمة"**: لا يوجد أي رابط تنقّل إليها من أي قائمة في أي لوحة تحكم.
- **جدول أسماء السور ناقص**: يحتوي على أول 20 سورة فقط من 114، فباقي السور تظهر بتسمية عامة "سورة رقم X".
- **قسم "المعلمين" بالصفحة الرئيسية بيانات ثابتة** (Hardcoded) وليس متصلًا بواجهة دليل المعلمين العامة الحقيقية.
- **حقول SMTP وZoom في إعدادات الأكاديمية غير مستخدَمة فعليًا** — مخزّنة فقط، بلا أي كود يقرأها.
- **لا يوجد ملف تهيئة ESLint في المشروع** — أداة فحص الكود مُثبَّتة لكن غير مُهيَّأة، فلا فحص جودة كود تلقائي فعلي حاليًا.

### ج) ميزات غير موجودة إطلاقًا (يجب توقّع تكلفة إضافية إن طُلبت)
- بوابة دفع إلكتروني حقيقية.
- تكامل فيديو حقيقي (Zoom/Meet API أو حل بث ذاتي).
- شهادات إتمام فعلية (يوجد فقط مفتاح تفعيل/تعطيل بلا أي توليد أو تحميل حقيقي).
- اختبارات/امتحانات (Quiz/Exam) بأي شكل.
- محادثة مباشرة (Chat) بين المستخدمين.
- دور ولي الأمر بالكامل.
- تحقق بريد إلكتروني و2FA.
- تسجيل دخول اجتماعي.
- تخزين سحابي للملفات.
- تطبيق موبايل (لا يوجد أي كود React Native/Flutter في المشروع).
- بنية تحتية للنشر (Docker/CI-CD).
